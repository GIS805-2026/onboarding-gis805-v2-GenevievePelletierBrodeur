# S00.5 — Explorer vos données avant d'écrire du SQL

Entre S00 (configuration) et S02 (première étoile), prenez 20-30 minutes
pour **vous familiariser avec les données** qu'on vous a générées. Sans
SQL — juste explorer et répondre à cinq questions.

## Pourquoi ?

Écrire un `fact_sales` sans savoir ce qu'il y a dans `raw_fact_sales`,
c'est comme écrire un résumé exécutif sans avoir lu le rapport. Les cinq
questions ci-dessous vous donnent la **connaissance du domaine** qui
transforme S02 d'un exercice de copie-collage en un exercice de décision.

## Pré-requis

- `make generate` exécuté (vos CSVs sont dans `data/synthetic/team_N/`).
- `make load` exécuté (les tables `raw_*` existent dans DuckDB).

## La boucle d'exploration

Ouvrez SQLTools dans VS Code ou lancez `duckdb db/nexamart.duckdb` dans
un terminal. Une cellule ou un onglet SQL, rien d'autre.

### Question 1 — Qu'est-ce qui est disponible ?

```sql
SHOW TABLES;
```

Vous devriez voir une dizaine de tables `raw_*`. Notez-en trois qui vous
intriguent.

### Question 2 — Combien y a-t-il de clients et de produits dans VOS données ?

```sql
SELECT COUNT(*) AS n_customers FROM raw_dim_customer;
SELECT COUNT(*) AS n_products  FROM raw_dim_product;
```

Comparez avec les fourchettes annoncées dans
[`docs/data-dictionary.md`](data-dictionary.md) (150-400 clients,
40-70 produits). Votre nombre est unique à votre team seed.

### Question 3 — À quoi ressemble une ligne de vente ?

```sql
SELECT * FROM raw_fact_sales LIMIT 5;
```

Pour chaque colonne, posez-vous : *est-ce un identifiant, une mesure, ou
un descripteur ?* Dites-le à voix haute (ou notez-le).

### Question 4 — Quelles sont les catégories de produits, et combien chacune pèse en revenu ?

```sql
SELECT
    p.category,
    COUNT(*)           AS n_lines,
    SUM(s.line_total)  AS revenue
FROM raw_fact_sales s
JOIN raw_dim_product p ON p.product_id = s.product_id
GROUP BY p.category
ORDER BY revenue DESC;
```

Vous venez d'écrire votre première requête dimensionnelle — avant même
d'avoir créé une étoile. Notez quelles catégories dominent chez votre
équipe : cette intuition vous sera utile pour le brief.

### Question 5 — Sur quelle période vos données s'étalent-elles ?

```sql
SELECT
    MIN(order_date) AS first_sale,
    MAX(order_date) AS last_sale,
    COUNT(DISTINCT order_date) AS n_distinct_days
FROM raw_fact_sales;
```

Vous aurez ~2 ans de ventes (2024-01-01 → 2025-12-31) mais pas forcément
une vente chaque jour. Les lacunes sont volontaires — elles reflètent la
réalité (dimanches fermés, fêtes, magasins peu fréquentés).

## Ce que vous écrivez

Ouvrez `answers/S00.5_exploration.md` (créez-le) et répondez en deux
paragraphes max :

1. **Ce que contient votre jeu de données** (3-4 chiffres clés, dans
   vos mots).
2. **Une observation intéressante** : un chiffre qui vous a surpris, une
   catégorie qui domine, une période qui semble anormale. Une phrase.

Pas de SQL obligatoire dans le fichier ; juste votre lecture de ce que
vous avez vu.

## Pourquoi ce livrable n'est pas noté

Parce qu'il est pour **vous**, pas pour le correcteur. Les étudiants qui
font cette exploration écrivent de meilleurs briefs en S02 parce qu'ils
ne raisonnent plus dans l'abstrait — ils raisonnent sur *leurs* chiffres.

## Astuce assistant IA

Collez cette commande et le résultat à votre assistant :

> J'ai exploré raw_fact_sales. Voici ce que je vois : [chiffres].
> Quelle hypothèse business pourrait expliquer la dominance de la
> catégorie X ? Donne-moi 2 explications plausibles que je pourrais
> vérifier par une autre requête.

Vous venez de transformer votre assistant IA en partenaire d'analyse,
pas juste en générateur de SQL.
